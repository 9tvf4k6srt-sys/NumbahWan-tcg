How It Works: Three Roles, One Pipeline

commit
→
LEARN
→
EVAL
→
FIX
→

→

Learner (Mycelium + Mycelium Watch)

Silently observes every commit. Tracks which files break, which change together, what areas need constraints. Zero config.

Evaluator (built into Fixer)

Scores both systems on 7 dimensions each. Finds weaknesses: repeat breakages, uncovered areas, shallow lessons.

Fixer (mycelium-fix.cjs)

Reads both brains, applies 8 fix actions, re-evaluates to verify. Loops up to 3 times until stable.

mycelium.cjs + mycelium-watch.cjs + mycelium-eval.cjs + mycelium-fix.cjs

2

Every commit triggers: learn → eval → fix → verify. No manual intervention ever needed.

How Mycelium Actually Helped This Project

NumbahWan started as a fun MapleStory guild site — 92 pages of satirical content, from fake stock exchanges to pixel museums. Then things started breaking. Here's the honest timeline of what happened when we added Mycelium, what worked, and what still doesn't.

Feb 3, 2026 — Day 0 (Before Mycelium)
The same files kept breaking
cards.html broke 10 times from the same i18n pattern. battle.html and its engine always changed together (28x) but nothing enforced it. We'd fix one page and break three others. No system tracked which files were coupled or what patterns caused failures.
Feb 4, 2026 — Mycelium installed
First scan: 150 commits analyzed in under 2 seconds
Mycelium read the entire git history and immediately found: 50 breakages, 200 file couplings, 15 recurring patterns. The pre-commit hook started warning about coupled files on the very first commit after install. No configuration was needed — it learned everything from git log.
Feb 5–7, 2026 — The eval system learned to be honest
Three rewrites of the scoring system
v1 scored 95/100 — clearly inflated. v2 added anti-gaming checks but still rewarded busywork. v3 (current) uses hash-locked thresholds: the score formula is deterministic, the self-checks include 4 anti-gaming rules, and 28/28 checks must pass. Score went from 95 to 75 — because that's the truth. Repeat Prevention is still 35/100 because 3 files still repeat.
Feb 8, 2026 — The upgrade tool generated real tests
52 regression tests auto-generated from breakage history
mycelium-upgrade.cjs read the 50 breakages, identified 15 repeat offenders, and generated targeted tests: 20 for i18n patterns, 9 for mobile issues, 3 for layout, 11 for structure, 9 for runtime. On first run, it caught 24 invalid i18n keys that existed in production.
Feb 9, 2026 — Mycelium caught a bug it learned about
GSAP defer issue caught by regression test
A performance optimization deferred all scripts across 63 pages — including GSAP CDN scripts in index.html that must run synchronously for the loading screen. A regression test (generated from a real breakage 5 days earlier) caught it before deploy. This is the loop working: break → learn → test → prevent.
Feb 10, 2026 — Deep lesson extraction added
Every commit now teaches the system, not just fixes
Before: only fix: commits extracted lessons (0/150 commits had insights). After: extractInsight() runs on every commit — refactor, feat, chore all get parsed for section headers and tags. 19/150 commits now have deep insights. The system learns from structural decisions, not just bugs.

What Mycelium can't do (honest limitations)

It doesn't write code. It tracks patterns and warns you, but you still need to fix the bugs yourself. The fixer suggests actions and can auto-generate tests, but it can't refactor your architecture.

Repeat Prevention is still 35/100. Three files (cards.html, markets.html, battle.html) keep breaking despite constraints. The system detects the pattern but can't force developers to follow the warnings.

It needs conventional commits to work best. If your commit messages are "update" or "fix", the lesson extraction is shallow. The system rewards specificity (fix(i18n): ...) because that's what creates actionable memory.

The score is 75, not 95. We could inflate it by padding metrics. We don't. The score drops when you do sloppy work (our score went to 55 after an 85-file sprawl commit). That's the point.

Why another dev should try this

  • 30-second install, zero dependencies. Copy 5 .cjs files → run node mycelium.cjs --init → done. No npm install, no config files, no CI setup needed.
  • It learns your codebase, not generic rules. ESLint checks syntax. Mycelium knows that your battle.html and nw-battle-engine.js break together because it read your git history.
  • Pre-commit hooks that actually help. Instead of "file too long" warnings, you get: "This file broke 10x from i18n issues — last time was 3 days ago. The coupled file (nw-nav.js) also needs updating."
  • Auto-generated regression tests from real breakages. Not random fuzz tests — tests that target the exact patterns that caused your past bugs. Run node mycelium-upgrade.cjs --apply and get tests written from your own history.
  • Honest scoring with anti-gaming. The eval system has 28 self-checks including 4 anti-gaming rules. You can't inflate the score without actually improving the codebase. SHA-256 proof on every evaluation.
  • Works on any git repo. Vanilla JS, TypeScript, Python, Go — if it has git commits, Mycelium can learn from it. No framework dependency.
